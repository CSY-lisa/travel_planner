### 問題 A：clasp push 報錯 `Request contains an invalid argument`

**狀況：**
- Script ID 已填入 `gas/.clasp.json`（`1TdcphxH...`）
- `clasp push` 報錯：`Request contains an invalid argument`
- 原因：本地 `gas/appsscript.json` 是我們新建的，與 GAS 雲端現有設定不符

**不能用 `clasp pull` 的原因：**
- `clasp pull` 會把 GAS 雲端的 `.gs` 檔案覆蓋到本地
- 本地的 `.gs` 才是最新版，不能被覆蓋

**解法（下次從這裡開始）：**

**Step A-1：只取得正確的 appsscript.json，不動 .gs 檔**
```bash
cd ".../travel_planner/gas"

# pull 下來（會覆蓋 .gs，沒關係，下一步馬上還原）
clasp pull

# 立刻用 git 還原本地最新的 .gs 檔案（appsscript.json 不還原，保留 GAS 的版本）
git checkout -- Code.gs GeminiService.gs LineService.gs SheetsService.gs
```

**Step A-2：確認 appsscript.json 內容後推送**
```bash
cat appsscript.json   # 確認內容看起來正常
clasp push            # 這次應該成功
```

**成功標誌：** 看到 `Pushed 5 files.`（4 個 .gs + 1 個 appsscript.json）

### 問題 B：GitHub Actions 的 pull 與 push 衝突邏輯

**狀況與疑問：**
你在問：「為什麼 GitHub Actions 要先 `git pull` 才能 `git push`？這樣不會覆蓋我本地的版本嗎？」

**釐清：這裡有兩個完全不同的 pull，不要混淆**

| 指令 | 是誰在跑 | 動到的東西 | 會不會影響你本地？ |
|------|----------|-----------|-------------------|
| `git pull`（GitHub Actions 裡）| GitHub 伺服器上的機器人 | 伺服器上的檔案 | ❌ 完全不影響你本地 |
| `clasp pull`（你在終端機跑）| 你的本地電腦 | 你本地的 `.gs` 檔案 | ✅ 會覆蓋本地 |

**GitHub Actions 的 `git pull` 是這樣運作的：**
```
[GitHub Actions 機器人，在遠端伺服器上]
1. git pull   ← 機器人同步自己，不動你本地
2. 更新資料（抓 Sheets 資料寫入 JSON）
3. git push   ← 把新資料推回 GitHub
```

**你本地的版本永遠安全，因為：**
- GitHub Actions 跑在 GitHub 的雲端伺服器，跟你電腦完全無關
- 你本地要同步最新資料，只需要 `git pull`（你自己決定要不要拉）
- 你的 `.gs` 程式碼不在 GitHub Actions 的更新範圍內，它只更新 `data/*.json`

**如果你本地和 GitHub 之間出現衝突：**
```bash
# 查看衝突原因
git status
git log --oneline -5

# 安全的同步方式（保留你本地的版本為主）
git pull origin main --rebase
git push
```

### 🔲 C. 旅程規劃 Skill（project level）

**目標：** 建立一支 project-level skill，讓 AI 能根據 `column_definition.json` schema 自動規劃並寫入 Google Sheets 行程。

**Skill 核心流程：**
1. 讀取 `data/column_definition.json` 理解每欄邏輯
2. 詢問用戶確認：
   - 時間區間（起訖日期）
   - 人數
   - 交通方式（自駕 / 大眾運輸）
   - 旅遊偏好（古蹟、自然、美食、文青…）
   - 旅遊步調（緊湊 / 休閒）
   - 是否有老人/小孩同行（影響休息安排）
   - 最早出發時間（預設 8–9 點）
   - 其他備註需求
3. 排程大原則：
   - 早上一大景點、下午一大景點
   - 預留 1 小時午餐、30–60 分下午茶
   - 所有景點以住宿點出發**單程 1.5 小時內**（來回 3 小時）
   - 以交通便利性為主，減少無謂移動
4. 先產出景點骨架給用戶確認（每天建議景點＋原因）
5. 用戶確認後 → web search 每個景點最新資訊 → 填滿所有欄位 → 寫入 Sheets

**執行方式：** 下次對話先 `superpowers:brainstorming`，再 `superpowers:writing-plans`，最後用 skill-creator 建立。
